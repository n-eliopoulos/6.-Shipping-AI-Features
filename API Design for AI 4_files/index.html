Timeouts and Long-Running Inference

Default 30-second timeouts will kill your AI calls. A complex compliance analysis can take 45 seconds. You need a layered timeout strategy: client timeout, server timeout, and model timeout — each serving a different purpose.

The timeout stack

LayerDefaultAI RecommendationPurpose
Client (browser/caller)30s60–120sHow long the caller waits before giving up
Server (API gateway)30s90sHow long the server holds the request
Model (Bedrock call)60s60sHow long the model provider allows

The graceful timeout pattern

async def analyze_with_timeout(request, timeout=10):
    try:
        result = await asyncio.wait_for(
            call_model(request), timeout=timeout
        )
        return {"status": "complete", "result": result}
    except asyncio.TimeoutError:
        job_id = enqueue_background(request)
        return {
            "status": "processing",
            "job_id": job_id,
            "message": "Analysis is taking longer than usual. Poll for results."
        }

Circuit breaker basics

When a model is consistently slow or failing, stop sending requests. A circuit breaker tracks failures and "opens" the circuit after a threshold — returning a fast fallback instead of waiting for another timeout.

Try it now: In the sandbox, configure timeout handling for an AI call. Simulate a 15-second response and implement a timeout wrapper that returns a graceful "still processing" response at 10 seconds.
Never use default timeouts for AI endpoints. Set explicit timeouts at every layer, and always have a plan for what happens when the timeout fires — don't just return a 504.
