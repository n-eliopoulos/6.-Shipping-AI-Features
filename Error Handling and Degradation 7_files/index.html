Module 4: Error Handling and Degradation

Exercise: Build a Resilient AI Feature

Build a regulatory report generation feature with full error handling. This is the capstone for the course — every pattern you've learned comes together here.

Requirements

  1. 3-tier model fallback — Sonnet → Haiku → cached response
  2. Rate limiting with queuing — 10 requests/minute, queue overflow, no drops
  3. Circuit breaker — Open after 3 failures, half-open after 30s, close after 2 successes
  4. Timeout handling with streaming — 10s timeout, graceful "still processing" response
  5. Budget enforcement — Daily budget with warning/critical/hard stop thresholds
  6. User-friendly error messages — Every failure mode produces an appropriate message

Chaos test suite (10 scenarios)

  1. Normal request — succeeds with primary model
  2. Rate limit — request queued and eventually processed
  3. Single timeout — graceful "still processing" response
  4. Primary model failure — falls back to Haiku
  5. Both models fail — returns cached response
  6. 3 consecutive failures — circuit breaker opens
  7. Circuit breaker recovery — half-open → probe → close
  8. Budget warning — alert fires at 80%
  9. Budget exhaustion — hard stop with graceful message
  10. Combined: rate limit + timeout + fallback in sequence

Validation

All 10 scenarios must pass: no unhandled exceptions, no silent failures, every failure produces an appropriate user-facing message, fallback chain activates correctly, budget hard stop works.

Production AI features fail in production. The question isn't whether they'll fail — it's whether your system handles failure gracefully or crashes spectacularly. Build for failure from the start.
