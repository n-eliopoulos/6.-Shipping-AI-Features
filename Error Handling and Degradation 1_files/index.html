AI Failure Modes

AI systems fail in ways traditional systems don't. Each failure mode requires a different response. Designing for failure starts with knowing the taxonomy.

Failure mode taxonomy

FailureHTTP codeCauseCorrect response
Rate limit429Too many requests to BedrockQueue and retry with backoff
Model timeout504Inference exceeds time limitRetry once, then fallback to simpler model
API outage503Bedrock service unavailableCircuit breaker → cached response
Malformed response200 (bad body)Model returns unparseable outputRetry with stricter prompt, then fallback
Hallucination200 (wrong content)Model generates false informationValidation layer, confidence thresholds
Context overflow400Input exceeds context windowTruncate or chunk input
Budget exhaustionN/A (internal)Daily/monthly budget exceededReturn cached result + notification

The key insight

Traditional error handling treats all failures the same: retry, then fail. AI error handling must differentiate — a rate limit needs queuing, a hallucination needs validation, a budget exhaustion needs a completely different code path.

Map every failure mode to a specific response before you deploy. If your error handling is a single except Exception block, your AI feature is not production-ready.
