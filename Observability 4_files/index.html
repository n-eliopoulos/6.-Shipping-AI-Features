Module 2: Observability

Trace Aggregation

A single user request may trigger 3 AI calls: classify → retrieve → summarize. Tracing connects them into a single story. Without traces, you see isolated calls. With traces, you see the full workflow — timing, dependencies, and where things went wrong.

Trace anatomy

Trace: workflow-abc-123 (total: 8.2s, cost: $0.034)
├── span: classify    (1.2s, 450 in / 80 out,  $0.002)
├── span: retrieve    (0.3s, — no AI call, vector search —)
└── span: summarize   (6.7s, 2100 in / 520 out, $0.014)

Key concepts

  • Trace ID. A unique identifier for the entire workflow. Propagated through every step.
  • Span. One unit of work within a trace. Each AI call is a span. Non-AI steps (DB queries, vector search) are also spans.
  • Parent/child hierarchy. Spans nest. The top-level request is the root span; each AI call is a child.
  • Timing waterfall. Visual representation showing when each span started, how long it took, and which ran in parallel.

What traces reveal

  • Which step in a multi-call workflow is the bottleneck
  • Whether steps could run in parallel instead of sequentially
  • Total cost of a user request across all AI calls
  • Where errors propagate through the workflow

Traces turn "the AI is slow" into "the summarize step takes 6.7 seconds because the input is 2,100 tokens — let's trim the context." Without traces, you're guessing.
