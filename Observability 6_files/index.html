Alerting on AI Metrics

Metrics without alerts are just dashboards nobody checks. Define alert rules for the conditions that require human action — and tune thresholds so they're not too noisy to be useful.

What to alert on

ConditionThreshold exampleAction
Latency spikep95 > 10s for 5 minutesCheck model status, consider fallback
Error rate increase>5% errors in 10-minute windowOpen circuit breaker, page on-call
Cost anomalyHourly cost > 2× rolling averageInvestigate prompt injection or runaway client
Token usage surgeOutput tokens > 3× expected per callCheck for prompt injection causing verbose output

Static thresholds vs. anomaly detection

  • Static thresholds — Simple, predictable. "Alert if p95 latency > 10s." Works well for known baselines.
  • Anomaly detection — Learns normal patterns, alerts on deviations. Better for cost and token usage where "normal" varies by time of day.
Try it now: In the sandbox, configure alert rules for the metrics dashboard. Then simulate a latency spike (model slowdown) and a cost anomaly (prompt injection causing huge outputs). Verify your alerts fire correctly.
Start with 4 alerts: latency spike, error rate increase, cost anomaly, and token usage surge. Tune thresholds after a week of baseline data. An alert that fires 10 times a day gets ignored.
