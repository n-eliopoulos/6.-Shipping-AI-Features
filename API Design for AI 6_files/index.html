Module 1: API Design for AI

Authentication for AI Services: The Concept Primer

Every production AI feature has the same two questions to answer about every incoming request: who is this? (authentication) and what are they allowed to do? (authorization). API keys handle these for development; production needs more.

Why API keys aren't enough at FINRA scale

  • No user identity — a single key represents an entire service. You can't tell which user triggered which call. Audit trails dead-end at the service.
  • No expiry — keys live until manually rotated. A leaked key keeps working until someone notices.
  • No fine-grained scope — a key either works for the service or doesn't. You can't say "this caller can read but not write."
  • No federation — every service rolls its own key store. Adding or removing a person means N service-team tickets.

OAuth 2.0 and OpenID Connect (OIDC) — the production answer

OAuth 2.0 is a delegation protocol: the user authenticates with a trusted identity provider, which then issues short-lived access tokens that services can verify. OpenID Connect (OIDC) sits on top of OAuth 2.0 and adds an ID token — a signed JWT containing identity claims (who the user is, which groups they belong to, when the session expires).

FINRA's identity provider is FIP. Every production AI service authenticates users against FIP using OIDC and validates the resulting tokens server-side. There's no per-service password store; FIP is the single source of truth.

The five things a JWT carries

  1. Subject (sub) — the user's identifier (e.g., jdoe@finra.org).
  2. Issuer (iss) — which identity provider minted the token (FIP).
  3. Audience (aud) — which service the token is intended for. A service must reject tokens issued for other audiences.
  4. Expiration (exp) — the unix timestamp when the token stops being valid. Typically ~1 hour.
  5. Claims — additional properties: scopes, AD groups, custom attributes. These drive authorization decisions.
// A real (but redacted) FIP-issued ID token, decoded:
{
  "sub": "jdoe@finra.org",
  "iss": "https://isso.fip.finra.org/.../isso",
  "aud": "fillip-prod",
  "exp": 1718550000,
  "iat": 1718546400,
  "nonce": "9d4b8a12...",
  "scope": "openid finra_profile FILLIP",
  "groups": ["FILLIP_USERS", "TRADING_ANALYSTS"]
}

OIDC turns "who is this?" into a token-validation problem instead of a credential-storage problem. Your service never sees the user's password — it just verifies a signed JWT against the identity provider's public keys.
