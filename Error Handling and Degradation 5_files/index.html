Module 4: Error Handling and Degradation

User-Facing Error Design

Users don't care about 429s or 503s. They care about: "Is my request being processed?" Error messages should be honest, helpful, and actionable — not technical status codes.

Bad vs. good error messages

FailureBadGood
Rate limit"Error 429: Too Many Requests""High demand right now. Your request is queued — estimated wait: 30 seconds."
Timeout"Internal Server Error""Analysis is taking longer than usual. We'll notify you when it's ready."
Model outage"Service Unavailable""We're using a simplified analysis while our primary system recovers. Full results available soon."
Budget exhaustion"Request Denied""Daily analysis limit reached. Cached results are available. Full service resumes tomorrow."
Slow response(blank screen for 10s)Progress indicator: "Analyzing trade patterns... Assessing risk factors... Generating report..."

Error message principles

  • Honest: Don't pretend everything is fine. "Using simplified analysis" is better than silently degrading.
  • Helpful: Tell the user what's happening and what to expect next.
  • Actionable: Give the user something to do — wait, retry, use cached results, or contact support.
Interactive: The error UX gallery below shows side-by-side comparisons of bad vs. good error handling for 5 AI failure scenarios. Notice how the good versions always tell the user what's happening and what to do next.
Every error message should answer three questions: What happened? What are we doing about it? What can the user do? If your error message doesn't answer all three, rewrite it.
