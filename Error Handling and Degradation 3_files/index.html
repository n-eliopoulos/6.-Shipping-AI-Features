Rate Limit Management

Bedrock enforces rate limits per account: requests per minute and tokens per minute. When you hit them, you get 429 errors. The solution isn't to retry blindly — it's to manage your request flow.

Understanding rate limit headers

HTTP/1.1 429 Too Many Requests
Retry-After: 2
X-RateLimit-Limit-Requests: 100
X-RateLimit-Remaining-Requests: 0
X-RateLimit-Reset-Requests: 2s

Token bucket implementation

class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate  # tokens per second
        self.last_refill = time.time()

    def consume(self, tokens=1):
        self._refill()
        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        return False

    def _refill(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity,
                         self.tokens + elapsed * self.refill_rate)
        self.last_refill = now

Request prioritization

Not all AI calls are equal. When capacity is limited, prioritize:

  1. User-facing requests (real-time analysis)
  2. Time-sensitive batch jobs (regulatory deadlines)
  3. Background processing (report generation)
Try it now: In the sandbox, build a rate limiter for AI calls. Configure at 10 requests/minute. Submit 20 requests. Verify: first 10 succeed immediately, next 10 queue and execute as capacity frees. No requests dropped.
Rate limiting is about flow control, not rejection. Queue requests, prioritize by importance, and spread load across time windows. Dropping requests should be the last resort.
