Endpoint Design Patterns

AI endpoints fall into three patterns: synchronous (simple, blocking), asynchronous (poll for results), and streaming (token-by-token). The right choice depends on latency tolerance and response size.

Synchronous — request/response

POST /api/v1/analyze
{
  "trade_description": "Large block trade in AAPL...",
  "analysis_type": "risk_assessment"
}
→ 200 OK
{
  "risk_level": "elevated",
  "confidence": 0.87,
  "citations": ["Rule 5210", "FINRA Notice 21-03"]
}

Best for: fast tasks (<5s), backend pipelines where the caller can wait.

Asynchronous — submit + poll

POST /api/v1/analyze → 202 Accepted { "job_id": "abc-123" }
GET  /api/v1/jobs/abc-123 → 200 { "status": "processing" }
GET  /api/v1/jobs/abc-123 → 200 { "status": "complete", "result": {...} }

Best for: long-running analysis (>10s), batch processing, workflows where the user navigates away.

Streaming — Server-Sent Events

POST /api/v1/analyze?stream=true
→ 200 OK (text/event-stream)
data: {"token": "The"}
data: {"token": " trade"}
data: {"token": " shows"}
data: {"done": true, "usage": {"input_tokens": 450, "output_tokens": 128}}

Best for: user-facing chat, long-form generation, any case where perceived latency matters.

Try it now: In the sandbox, build a request/response schema for a trade surveillance /analyze endpoint. Define the request body (trade description, analysis type) and response body (risk assessment, confidence, citations).
Most AI features need at least two patterns: synchronous for simple calls, and either async or streaming for longer ones. Design both from the start — retrofitting async is painful.
