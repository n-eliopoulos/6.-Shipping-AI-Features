OIDC Walkthrough: What Happens When a User Logs In

OIDC is built on the authorization code flow. The user, your service, and the identity provider exchange a sequence of redirects and tokens. Knowing the shape of this flow is enough at the concept level — you'll implement it at depth in the CI/CD & Application Security course.

The eight-step handshake

  1. User hits your service. No session cookie yet — your service responds 401 (or redirects to login).
  2. Your service redirects to FIP with parameters identifying itself (client_id), where to come back (redirect_uri), what's being requested (scope=openid finra_profile), and three security values: state (CSRF protection), nonce (replay protection), and code_challenge (PKCE).
  3. FIP authenticates the user — credentials, MFA, whatever its policy says. This part is FIP's responsibility; your service is uninvolved.
  4. FIP redirects back to your service with an authorization code (one-time, ~60-second TTL) and the original state.
  5. Your service exchanges the code for tokens. A direct server-to-server POST to FIP including the code, the code_verifier (the original PKCE secret), and the service's own client credentials.
  6. FIP returns three tokens:
    • id_token — JWT with user identity claims.
    • access_token — used to call other services on the user's behalf.
    • refresh_token — long-lived, used to get new access tokens without re-authenticating.
  7. Your service validates the ID token: signature against FIP's JWKS, issuer, audience, expiration, and that the nonce matches what was sent in step 2. Anything off → reject.
  8. Your service creates a session for the authenticated user (typically a session ID in a server-side store, returned to the browser as an HttpOnly cookie). Tokens stay server-side; the browser never sees them.

The three layers of defense in this flow

LayerMechanismAttack prevented
CSRFstate parameterAttacker can't initiate a login on the victim's behalf — state mismatch on callback.
Token replaynonce in ID tokenAttacker can't reuse a stolen ID token in a new flow — nonce won't match.
Code interceptionPKCE (code_verifier)Attacker intercepting the authorization code from URLs/logs can't exchange it without the verifier.
