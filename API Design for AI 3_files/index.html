Module 1: API Design for AI

Streaming Responses

A 10-second wait with no feedback feels broken. Streaming sends tokens as they're produced — the user sees output from ~200ms instead of waiting for the full response.

Server-Sent Events (SSE)

The standard pattern for AI streaming. The server holds the connection open and pushes events as tokens are generated:

# Server side — SSE endpoint
def stream_analysis(request):
    yield "data: {\"token\": \"The\"}\n\n"
    yield "data: {\"token\": \" risk\"}\n\n"
    yield "data: {\"token\": \" level\"}\n\n"
    yield "data: {\"done\": true}\n\n"

# Client side — consuming the stream
const source = new EventSource("/api/v1/analyze?stream=true");
source.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data.done) source.close();
    else appendToken(data.token);
};

Streaming vs. non-streaming UX

MetricNon-streamingStreaming
Time to first visible content8 seconds~200ms
Total completion time8 seconds8 seconds
User perception"Is it broken?""It's working on it"
Try it now: In the sandbox, implement a streaming endpoint using ai_shim. Compare the UX of an 8-second blank wait vs. streaming tokens from 200ms.
Streaming doesn't make inference faster — it makes the experience faster. Use it for every user-facing AI feature. Skip it only for backend pipelines that need the complete response.
